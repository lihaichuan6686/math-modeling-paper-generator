\section{Data Processing}

The demo data are synthetic and are stored in the problem statement. Demand is given for eight weeks. Supplier information includes capacity, unit price, transport loss rate, and reliability. The purpose of this section is to make the v1.0 workflow behave like a contest paper: raw indicators are transformed into model parameters, and each transformation has a visible table or formula.

The demand series is:

\begin{equation}
(120,135,128,150,160,155,142,130).
\end{equation}

The data-cleaning checks used in this demo are deliberately simple:

\begin{enumerate}
\item all demand values must be positive;
\item all supplier capacities must be non-negative;
\item each loss rate must lie between 0 and 1;
\item each reliability score must lie between 0 and 1;
\item units are treated as consistent across demand, capacity, and inventory.
\end{enumerate}

For a real contest problem, this section would also include missing-value checks, abnormal supplier behavior, unit conversion, and feature construction. For v1.0, the data are already clean, so the processing step focuses on converting the supplier table into evaluation scores and planning parameters.

Table~\ref{tab:demo-v1-supplier-scores} records the processed supplier indicators and the final evaluation scores used by the ordering model. This table is generated by code and is therefore part of the reproducibility chain rather than a manually typed result.
