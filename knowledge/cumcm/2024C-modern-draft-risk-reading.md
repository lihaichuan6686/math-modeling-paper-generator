# 2024 C Modern-Draft Risk Reading: Partial, Plausible, and Overlong Outputs

Purpose: extend the modern-draft risk corpus from 2024 B to 2024 C and test whether claimed completeness, rendered length, and actual evidence stay aligned.

## Corpus

Problem: 2024 CUMCM C, crop-planting strategy.

Three locally collected complete-paper-labeled samples were inspected:

- set-3: 10 PDF pages, labeled as updated for question 1;
- set-6: 22 PDF pages, labeled as updated for questions 1-3;
- set-7: 69 PDF pages, labeled as a finished paper; the printed page number reaches 70 because of front-matter numbering.

The first three pages and selected final/code pages were rendered and visually inspected. These are reference/course-material drafts, not a verified official award corpus.

## Main Finding

The three samples expose three different failure modes:

```text
short template shell      -> coverage failure
reasonable-length draft  -> potentially usable route, still needs evidence audit
very long draft           -> length / appendix inflation risk
```

The generator must classify a paper by question coverage and reproducibility evidence, not by the filename, marketing label, or page count.

## Structural Lessons

The 2024 C problem supports a clear three-question progression:

1. allocate crops to land under rotation, area, and sales constraints;
2. optimize a multi-year plan under uncertain yield, cost, price, and demand;
3. add crop complementarity, substitution, and correlation effects.

For a useful paper, every question should preserve the chain:

```text
land / crop interpretation
-> decision variables and units
-> constraints and scenario policy
-> solver or optimization procedure
-> allocation table / objective value
-> sensitivity or scenario comparison
-> practical recommendation and limitation
```

The strongest-looking sample uses a recognizable modeling route: mixed-integer linear programming for the baseline allocation, scenario or stochastic optimization for uncertain future years, and a richer crop-interaction model for the final question. The route itself is plausible, but a long route description does not prove that the reported allocation was actually generated by the stated model.

## Risk Findings

### 1. Partial-paper labeling

The 10-page sample contains a generic title and abstract, empty question summaries, blank assumption items, and a notation table with headers but no entries. Its label explicitly says it covers only question 1. This is useful as a negative control: the pipeline should recognize it as partial even if a user calls it a paper.

Hard checks:

- extract the question numbers from the problem statement;
- require a result block for every required question;
- distinguish "mentioned" from "solved";
- reject an abstract containing generic `xx` / `xxxx` placeholders.

### 2. Length inflation

The 69-page PDF is far beyond the 20-30 page target. Its middle pages contain long code listings, repeated data-writing routines, and implementation detail that would usually belong in a supplementary artifact. The final rendered page still shows an appendix shell with an `xxxx` code label and no populated code block.

This establishes two independent gates:

- `paper_body_length`: the main argument remains within the target range;
- `reproducible_artifact_length`: code and data may be stored separately and referenced, but their page count cannot be used to inflate the paper body.

A code listing can be long and still fail if it is not connected to a run command, input file, output table, or model equation.

### 3. Code as decoration

The long sample includes real-looking MATLAB code, including Excel-writing calls and multi-year array operations. That is stronger than an empty box, but it still requires a provenance check. In particular, the reviewer should verify:

- the variables in code correspond to notation in the paper;
- the data dimensions match the stated years, plots, and crops;
- output files named in code exist after execution;
- the reported allocation tables can be regenerated;
- the code does not merely export precomputed arrays.

The generator should never award full reproducibility credit solely because a code block contains many lines.

### 4. Watermark and template residue

The samples carry repeated promotional banners and large translucent watermarks. Some pages also retain template headings and empty tables. These residues harm readability and make the output unsuitable as a clean research artifact.

The rendered-PDF review must detect both visible residue and source-level residue. A clean output should keep provenance in a machine-readable manifest, not in page overlays.

### 5. Uncertainty claims need scenario evidence

The plausible 22-page route mentions forecast growth ranges, price and yield fluctuations, and random optimization. To make these claims auditable, the paper needs a scenario table or figure showing at least:

- the uncertainty ranges or distributions;
- the number or construction rule of scenarios;
- the decision policy used across scenarios;
- objective values and allocation changes;
- sensitivity to the most influential assumptions.

Saying "Monte Carlo" or "random programming" in the abstract is not enough. The generated paper must expose the uncertainty-to-decision mapping.

## Generator Gates Added From This Comparison

1. `coverage_gate`: match required questions to non-empty model/result sections.
2. `page_budget_gate`: keep the main paper near 20-30 pages; move bulky code to an artifact folder.
3. `printed_pdf_gate`: compare PDF page count with visible page numbers and detect front-matter offsets.
4. `placeholder_gate`: reject `xx`, `xxxx`, empty numbered assumptions, empty notation rows, and generic appendix labels.
5. `code_link_gate`: map each code module to an equation, input, output, and execution command.
6. `scenario_gate`: require explicit uncertainty ranges, scenario construction, and result comparison.
7. `allocation_gate`: require populated crop-area tables with units, land blocks, years, and rotation constraints.
8. `appendix_gate`: an appendix heading without an artifact is a hard failure.
9. `watermark_gate`: reject promotional overlays and repeated distribution banners in final rendering.
10. `claim_gate`: do not claim global optimality, robustness, or practical superiority without a baseline or sensitivity result.

## Implications For The 20-30 Page Blueprint

The C route suggests the following evidence-first allocation:

- 2 pages: problem and data interpretation;
- 2 pages: assumptions, units, and decision-variable ledger;
- 4-5 pages: baseline land-allocation model and results;
- 4-5 pages: uncertain multi-year model, scenario design, and comparison;
- 3-4 pages: crop interaction / substitution extension;
- 2-3 pages: sensitivity, limitations, and recommendations;
- 2-3 pages: abstract, references, and a concise reproducibility note.

The full source code should be delivered alongside the paper, with a short paper appendix containing only the key pseudocode or entry points. This keeps the paper readable without weakening reproducibility.

## Review Questions

- Does the paper solve all required questions, or only mention them?
- Is the 20-30 page target measured on the paper body rather than code dumps?
- Can the area allocations be regenerated from the stated data and constraints?
- Are future-year recommendations a policy over scenarios or just one arbitrary forecast?
- Does the final question add an identifiable interaction mechanism rather than a new name for the same model?
- Are code, data, and output tables linked by stable filenames and commands?

## Status

This note is a risk-reading extension. It should be combined with official excellent-paper reads and same-problem comparisons; it is not an endorsement of the source drafts.
