## Core Soft Rules

### 1. The Paper Is The Only Judging Artifact

Experience notes emphasize that the written answer is the only basis for judging.

Therefore the generator must optimize:

- clarity;
- traceability;
- visible results;
- concise reasoning;
- polished expression.

Good code without visible paper evidence does not count.

### 2. The Abstract Is A Competition Surface

The abstract should not merely introduce methods.

It should state:

- model type or route;
- modeling idea;
- solving idea;
- model advantages or validation;
- main numeric results and answers to every subquestion.

If the abstract does not answer the tasks, the paper feels weak even when the body is decent.

### 3. Do Not Force A Problem Into A Reference Template

Training material warns that many teams take reference material and force the problem into it.

The generator should treat references as options, not molds.

Route choice must come from:

```text
problem object
-> decision object
-> evidence burden
-> available data
```

### 4. Simple, Practical, Explainable Methods Are Often Stronger

Good modeling does not mean high-level terminology.

Prefer:

- simple baseline;
- clear improvement;
- visible comparison;
- practical interpretation.

Use advanced models only when they improve the result, evidence, or explanation.

Local experience notes repeat this in several forms: if an elementary or direct model already answers the task clearly, do not replace it with a less readable advanced method only for prestige. A stronger paper can still be innovative through a clean simplification, a sharper comparison, a useful validation step, or a better result display.

### 5. Exchange Ideas, But Do Not Lose The Route

Community experience says online discussion can inspire ideas, but it can also create panic.

The generator should use external discussion to:

- find common traps;
- compare rough result ranges;
- discover missing validation;
- sanity-check assumptions.

It should not chase every crowd answer.

### 6. Team-Like Papers Preserve Failed Or Rejected Ideas Selectively

One experience note says not every attempted solution should be thrown away.

In paper terms, this means:

- mention a rejected route when it explains why the final route is better;
- include baseline comparison when it clarifies improvement;
- do not clutter the paper with every dead end.

### 7. Do Not Compare Exact Answers Too Nervously

Some experience material warns that mathematical modeling does not have a single fixed answer.

Online or peer results should be used to check:

- whether the route misunderstands the task;
- whether the result scale is absurd;
- whether a key constraint was missed.

They should not cause blind route switching.

### 8. A Paper Needs A Center Thought

Community training material warns against stitching together reference screenshots or copied methods.

The generator should maintain one central route idea:

```text
problem interpretation
-> model route
-> result form
-> validation
-> final answer
```

Every method and figure should support that center.

### 9. Innovation Means Directness And Usefulness

Experience notes distinguish real innovation from fancy algorithm names.

Good innovation can appear as:

- a cleaner simplification;
- a better result representation;
- a useful comparison;
- a practical validation;
- a more direct route to the real decision.

Do not equate innovation with neural networks, swarm algorithms, or other prestige labels.

### 10. Results Must Be One-To-One With The Problem Statement

Every asked subquestion needs a visible answer.

Acceptable forms:

- result table;
- answer paragraph;
- decision list;
- final ranking;
- parameter table;
- predicted values;
- recommendation plan.

Unacceptable:

- only code output;
- result hidden in appendix;
- abstract claims without table support;
- conclusion that says "the model is effective" without the actual answer.

### 11. Search Experience Before Searching Answers

Local contest-experience notes encourage using networks and forums, but also warn that comparing exact answers can create panic.

The generator should search in this order:

```text
general contest experience
-> writing and structure rules
-> domain background
-> exact problem discussion only after route-decision
```

This keeps online material in the role of reflection, not route ownership.

### 12. Section Duty Matters More Than Section Name

The local excellent-paper reading note evaluates each section by function:

- abstract gives methods and key conclusions;
- problem restatement clarifies the real task;
- problem analysis names conditions and route;
- assumptions make the problem solvable and are used later;
- symbols reduce later ambiguity;
- model solution connects formulas, code, results, and interpretation;
- error analysis explains the most arguable places;
- model evaluation states strengths and concrete limits;
- references support outside claims;
- appendix supports reproduction.

A v1.4 paper fails if these sections exist only as headings.

When reading or imitating an excellent paper, extract section behavior rather than surface wording. For each section ask:

- what problem duty did this section perform;
- what evidence or figure made the duty visible;
- what weakness did the reader notice;
- what rule should prevent that weakness in a new run.

Do not copy a winning paper's route blindly. The reusable value is often the section logic: concise restatement, route-aware analysis, assumptions that enable the model, symbols that actually reappear, body-visible answers, error analysis around the most arguable choice, and appendix code after the paper has already answered the task.

### 13. Strong-Team Feel Comes From Division Of Labor Becoming Visible

Experience notes often describe one teammate focusing on modeling, one on implementation, and one on writing.

The paper should look like that collaboration happened:

- model sections carry mathematical route logic;
- code outputs become clean tables and figures;
- writing connects results back to decisions;
- review sections catch contradictions before final delivery.

If the draft reads like one stream of generic text, it loses human-team credibility.

## Practical Run Rules

- Read the problem like a reviewer, not like an algorithm catalog.
- Decide what must be filled in, ranked, predicted, or recommended.
- Build at least one simple route before adding complex methods.
- Treat advanced algorithms as upgrades that must justify what they improve.
- Make figures and tables argumentative, not decorative.
- Keep problem restatement short.
- Make problem analysis route-aware.
- Write validation as evidence, not ceremony.
- Make limitations specific and non-dramatic.
- Keep one center thought for the whole paper.
- Use online discussion for reflection, not panic.
- Search public experience before exact problem answers.
- Make each section perform a real duty.
- Let modeling, coding, writing, and review all leave visible evidence in the final paper.
