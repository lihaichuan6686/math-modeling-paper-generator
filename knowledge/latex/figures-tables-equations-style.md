# Figures, Tables, and Equations Style Guide

Purpose: make generated CUMCM papers visually and structurally credible. Figures, tables, and equations are not decoration; they carry modeling evidence.

## File Organization

Recommended paths:

```text
paper/figures/
paper/tables/
src/figures/
src/tables/
runs/current/artifact-ledger.md
runs/current/figure-plan.md
```

Rules:

- Use stable English filenames.
- Avoid spaces and Chinese characters in figure/table filenames.
- Data-driven figures should be generated by scripts.
- Explanatory schematic figures must record source or prompt.
- Do not manually paste screenshots unless the source is recorded.

Good filenames:

```text
fig_route_overview.pdf
fig_supply_capacity_curve.png
fig_model1_sensitivity_temperature.pdf
tab_supplier_scores.tex
tab_model_parameters.csv
```

Weak filenames:

```text
1.png
result-final-new.png
微信图片.png
```

## Figure Rules

Every figure must have:

- file path
- caption
- label
- text reference
- artifact-ledger entry

Preferred LaTeX pattern:

```latex
\begin{figure}[!htbp]
  \centering
  \includegraphics[width=0.82\textwidth]{fig_route_overview}
  \caption{Overall modeling route from task decomposition to validation.}
  \label{fig:route-overview}
\end{figure}
```

Caption rule:

```text
object shown + modeling purpose + interpretation cue
```

Examples:

- Weak: `Prediction result.`
- Strong: `Rolling prediction of weekly demand under Model II. The model captures the seasonal trend, while residual peaks identify holiday-week underestimation.`

Recommended width:

- single figure: `0.75\textwidth` to `0.9\textwidth`
- small side-by-side figures: `0.45\textwidth`
- full-width complex plots: `\textwidth`

Do not:

- insert unlabeled figures
- insert a figure without discussion
- use low-resolution images for charts
- claim an AI-generated schematic is measured data
- borrow raw one-off local filenames such as `output3.png` or `5-2.png` into generator output naming

## Subfigure Rules

Use subfigures when comparing:

- before/after
- baseline/model
- different scenarios
- multiple feature projections

Pattern:

```latex
\begin{figure}[!htbp]
  \centering
  \begin{subfigure}{0.48\textwidth}
    \centering
    \includegraphics[width=\textwidth]{fig_before}
    \caption{Before adjustment}
    \label{fig:before}
  \end{subfigure}
  \begin{subfigure}{0.48\textwidth}
    \centering
    \includegraphics[width=\textwidth]{fig_after}
    \caption{After adjustment}
    \label{fig:after}
  \end{subfigure}
  \caption{Comparison of the system state before and after optimization.}
  \label{fig:before-after}
\end{figure}
```

## Table Rules

Every table must have:

- caption
- label
- units
- reasonable precision
- text reference
- artifact-ledger entry

Preferred three-line table:

```latex
\begin{table}[!htbp]
  \centering
  \caption{Main parameter settings for Model I}
  \label{tab:model1-params}
  \begin{tabular}{lll}
    \toprule
    Parameter & Meaning & Value \\
    \midrule
    $N$ & Number of samples & 1000 \\
    $\epsilon$ & Error tolerance & 0.01 \\
    \bottomrule
  \end{tabular}
\end{table}
```

Rules:

- Use `booktabs` style.
- Avoid vertical lines in standard result tables.
- Use consistent decimal places.
- Put units in headers.
- For long tables, use `longtable` or move details to appendix.
- For wide tables, reduce columns or rotate/move to appendix; do not let the PDF clip.

Local contest note:

- awarded local samples may still use bordered tables with vertical rules; treat that as a tolerated local habit, not as the default generator style.

## Equation Rules

Every important equation must:

- define variables nearby or in the symbol table
- have a label when referenced
- be connected to the problem or algorithm
- not introduce unused intermediate variables

Pattern:

```latex
\begin{equation}
  \min Z = \sum_{i \in I}\sum_{j \in J} c_{ij}x_{ij}
  \label{eq:transport-objective}
\end{equation}
```

Then reference:

```latex
Equation~\eqref{eq:transport-objective} minimizes total transportation cost.
```

For constraints:

```latex
\begin{align}
  \sum_{j \in J} x_{ij} &\leq s_i, && i \in I, \label{eq:supply}\\
  \sum_{i \in I} x_{ij} &\geq d_j, && j \in J, \label{eq:demand}\\
  x_{ij} &\geq 0, && i \in I, j \in J. \label{eq:nonnegative}
\end{align}
```

Do not:

- add decorative formulas that are never used
- reference equations by manual numbers
- use different symbols in code, table, and equation without mapping

## Algorithm Description

Use a step list or pseudo-code when algorithm logic matters.

Minimum content:

- input
- output
- initialization
- iteration/update rule
- termination condition
- parameter values

Simple pattern:

```text
Input: cleaned data matrix X, parameter set Theta
Output: optimized decision vector x*
Step 1: initialize ...
Step 2: evaluate ...
Step 3: update ...
Step 4: stop when ...
```

Avoid:

- pasting long code into the body
- giving only package or solver names
- omitting search ranges and stopping criteria

## Appendix Code Rules

The appendix should support reproducibility, but the runnable source must remain in `src/`.

Appendix should include:

- file inventory
- environment notes
- key code excerpts
- random seeds
- command examples
- extra results

Avoid:

- putting the only copy of code inside the PDF
- dumping hundreds of lines without explanation
- including unrelated template code

## PDF Inspection Checklist

After compiling:

- figures are not missing
- figures are not blurry
- tables are not clipped
- equation numbers are correct
- references are not broken
- captions are readable
- no stale template identity fields remain
- page count matches the expected structure

If a figure/table causes layout problems:

1. shrink only within readability limits
2. split into multiple figures/tables
3. move detailed versions to appendix
4. keep summary version in the body
