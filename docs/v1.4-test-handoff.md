# v1.4 Test Handoff

Purpose: give the user the shortest practical handoff for testing v1.4 in Claude Code.

Use this when the repository has been cloned or pulled and the user wants to test whether v1.4 can generate a more human-team-like mathematical modeling paper.

## Before The Test

From the repository root, run the static check:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-v1.4-static.ps1
```

The check must pass before treating the repository as a v1.4 test candidate.

Prepare:

- one fresh problem statement;
- all attachments and data files;
- any required LaTeX or Word template;
- optional excellent-paper comparison sample.

## Prompt To Claude Code

Use this user message in Claude Code after the repository is available:

```text
Use the repository skill at .codex/skills/cumcm-paper-generator/SKILL.md.
Read START_HERE.md and CLAUDE.md.
Use prompts/12_launch_v1_4.md as the main instruction.

Generate a v1.4 CUMCM-style mathematical modeling paper draft for the active problem.

Requirements:
- build the run scaffold and planning files first;
- perform Playwright MCP public research after the initial model plan and before final writing;
- use prompts/13_platform_research.md before prompts/11_online_consensus_check.md when public-web access is available;
- use public sources only, do not copy answers, do not use cookies/login bypass/hidden APIs/internal endpoints, and record rejected or discounted signals;
- produce code-generated figures and tables where the problem supports them;
- make the abstract a dense solution surface with method-result closure for each major subquestion;
- keep final answer tables/figures in the body before appendix code;
- compile paper/main.pdf if local LaTeX is available;
- run prompts/06_quality_review.md and save the review.
```

## Expected Artifacts

At minimum, the test should produce:

- `runs/current/problem-profile.md`
- `runs/current/problem-analysis.md`
- `runs/current/route-decision.md`
- `runs/current/model-plan.md`
- `runs/current/online-consensus-notes.md`
- `runs/current/figure-plan.md`
- `runs/current/section-budget.md`
- `runs/current/writing-style-plan.md`
- `runs/current/artifact-ledger.md`
- `paper/main.tex`
- `paper/main.pdf` when compilation is available
- `reviews/review-current.md`

## Fast Pass/Fail

Mark the run as not v1.4-ready if any of these happen:

- no online-consensus note or skip reason;
- online research uses unsafe access methods or treats public popularity as proof;
- no body-visible final answer table/figure for a quantitative subquestion;
- abstract is short, method-only, or missing final answers;
- figures are decorative and not tied to subquestions;
- appendix/code carries the main answer instead of the body;
- review does not catch obvious missing evidence;
- final answers disagree across abstract, body, conclusion, and artifact ledger.

## Score The Run

Use:

```text
docs/v1.4-user-test-protocol.md
docs/v1.4-user-test-feedback-template.md
docs/v1.4-feedback-triage-matrix.md
```

The most useful feedback is:

- the generated `paper/main.pdf`;
- the quick score from the feedback template;
- the top three failures;
- whether `reviews/review-current.md` caught those failures;
- whether the paper feels closer to a human award-seeking team draft than v1.2.

After scoring, use `docs/v1.4-feedback-triage-matrix.md` to map the top failures to the next repository repair instead of starting a broad new version plan too early.

## What v1.4 Is Allowed To Be

v1.4 does not need to be competition-winning.

It should be visibly better than v1.2 in:

- abstract density;
- section duty;
- method-result-validation closure;
- figure/table evidence;
- online sanity reflection;
- human-team prose rhythm.

If these improve but the methods are still not award-competitive, that is a useful v1.4 test result and should feed v1.5.
