# v1.5 Feedback Triage Matrix

Purpose: turn a disappointing v1.5 generated paper into the next concrete repository repair without drifting into endless polish.

Use this after filling `docs/v1.5-user-test-feedback-template.md`.

## How To Use

1. Start with the most visible user-side failure.
2. Match it to the closest row below.
3. Inspect the evidence files before changing the repository.
4. Repair the smallest durable layer: prompt, template, checker, knowledge note, or run scaffold.
5. Run `scripts/check-v1.5-static.ps1`.
6. Retest with a fresh Claude Code run only after the repair is wired into the v1.5 entry path.

## Repair Layer Rule

| If the failure is... | Prefer repairing... |
|---|---|
| Claude forgot a required step | prompt or run scaffold |
| Claude followed the step but output looked wrong | template or knowledge rule |
| The failure should be automatically blocked | checker |
| The failure comes from weak contest feel | knowledge note plus template example |
| The failure needs real model correctness research | later-version backlog unless repeated |

## Repair Priority

When many failures appear, repair in this order:

1. title, abstract, and first-page visible failures;
2. method-route depth and validation artifacts;
3. result sanity and artifact traceability;
4. LaTeX template and appendix-code boundary;
5. figure/table readability and evidence role;
6. final review discipline;
7. environment/setup friction;
8. softer prose and contest-feel refinements.

## Stop Rule

Do not turn every complaint into a new rule.

Promote a repair into v1.5 only when at least one is true:

- it directly caused the generated paper to fail the user's first-look judgment;
- it can be checked or scaffolded reliably;
- it repairs a repeated failure across runs;
- it protects responsible-use, traceability, or evidence quality.

Otherwise record it in the feedback template and defer it to v1.6/v2.0 planning.
