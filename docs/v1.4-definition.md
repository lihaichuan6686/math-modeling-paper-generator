# v1.4 Definition

Purpose: define the v1.4 upgrade target for the CUMCM paper generator.

v1.4 is not an algorithm-depth release.

v1.4 is a contest-feel and community-soft-rule release.

It exists because many strong modeling papers win not only through model choice, but through:

- clear answer orientation;
- persuasive abstract and results;
- clean structure;
- visible comparison and validation;
- disciplined use of figures and tables;
- reviewer-friendly language;
- awareness of contest-circle soft rules.

## What v1.4 Adds

v1.4 adds four layers above v1.3:

1. community soft rules from modeling-experience material;
2. award-paper feel controls for abstract, section density, figures, tables, references, and prose;
3. an online-consensus-check protocol before final writing;
4. a human-feel review gate that checks whether the draft looks like a serious student team paper.

## What v1.4 Does Not Do

v1.4 should not:

- optimize one historical problem too deeply;
- create tiny route families such as one-off "spectral identification" rules;
- treat advanced algorithms as the main path to quality;
- prove the tool by local production tests.

Practice testing belongs to the user running Claude Code on real problems.
This repository's job is static learning, abstraction, and workflow improvement.

## Core Upgrade Principles

### 1. Learn The Circle, Not Only The Textbook

Experience notes repeatedly stress that the paper is the only judging artifact.

Therefore the generator must optimize the visible answer:

```text
what problem was solved
-> what model was used
-> what result was obtained
-> why the result is credible
-> what answer should be filled in
```

### 2. Simple And Useful Beats Decorative Complexity

Strong experience notes warn against forcing a problem into reference material or using advanced methods for show.

The generator should prefer:

- simple baseline first;
- useful improvement second;
- comparison and validation third;
- advanced method only when it improves the answer or explanation.

### 3. Results Must Be Easy To Find

Every required answer from the problem statement should appear in:

- abstract;
- result table or result paragraph;
- conclusion;
- artifact ledger.

If the paper answers a task only inside code, appendix, or vague prose, it fails v1.4.

### 4. External Discussion Is A Sanity Signal

Online discussions, forums, Bilibili videos, Xiaohongshu posts, and public articles can provide:

- common misunderstandings;
- rough result ranges;
- popular route choices;
- presentation tricks;
- warnings from former competitors.

They must not be copied as answers.
They should be used as a reflection layer before writing the paper.

## v1.4 Acceptance Standard

A v1.4-ready run must have:

- a v1.3 problem profile and route decision;
- a section budget that reflects award-paper feel;
- a writing-style plan that avoids machine-like section stacking;
- an online-consensus note, or an explicit reason why it was skipped;
- figure/table plans tied to each major subquestion;
- abstract and conclusion controls;
- a human-feel review record before handoff.

## Current Status

v1.4 begins from:

- v1.3 methodology scaffolds;
- local modeling-experience notes;
- existing official-paper reading notes;
- existing section and writing playbooks.

The next work is to strengthen static knowledge and prompts, not to continue single-problem modeling experiments.
