## v1.4 Workflow

```text
problem intake
-> v1.3 problem profile
-> v1.3 route decision
-> initial model plan
-> online consensus check
-> route and result sanity reflection
-> implementation and figures/tables
-> paper writing
-> v1.4 human-feel review
-> revision
```

## Key Timing Rule

Do the online-consensus check after an initial route exists but before final paper writing.

Too early:

- the model may copy noisy crowd direction;
- the agent may not know what to compare.

Too late:

- the paper may already be built around a wrong assumption;
- external signals cannot cheaply repair the route.

## Static Learning Rule

When improving the repository itself, prefer static learning:

- local experience notes;
- literature search and citation discipline;
- excellent-paper structure;
- abstract and section feel;
- common mistakes;
- table and figure density;
- reference and appendix habits;
- review language.

Do not spend repository-improvement time overfitting one rare topic.

## Online Consensus Rule

The consensus check is not answer copying.

It should record:

- what sources were searched;
- what common interpretations appeared;
- what rough result ranges or method families appeared;
- what looked unreliable or contradictory;
- what route changes, if any, were made;
- what was rejected and why.

Fail the note if it has no source-quality labels, no route reflection, no rejected signal, or treats a popular online method as proof.

## Paper Feel Rule

A v1.4 draft should look like a serious student team paper:

- abstract close to a full dense page when the task has multiple subquestions;
- abstract stays within the one-page ceiling while still giving method and result for each major subquestion;
- 20-30 pages are acceptable for a closed-loop v1 draft, while 30-45 pages often better matches recent award-paper visual rhythm when the evidence burden supports it;
- problem restatement short and model-oriented;
- problem analysis explains route, not background padding;
- assumptions, symbols, error analysis, evaluation, references, and appendix each perform a visible duty;
- pages 4-6 should already show symbols, tables, diagrams, equations, or model setup when the problem supports them;
- pages 8-12 should alternate artifacts and interpretation rather than remaining pure prose;
- the middle body should avoid more than 2-3 consecutive artifact-light pages in model-heavy problems;
- model section earns length through equations, artifacts, and interpretation;
- each major result has a nearby table/figure and explanation;
- validation is not one ceremonial paragraph;
- body result tables and figures appear before appendix code or support files;
- appendix code is mapped to subquestions and never replaces the main explanation;
- references support method, data, and domain claims without padding the paper;
- literature search produces route, validation, data, or citation changes rather than copied answer text;
- section proportions explain why the draft is a 20-30 page closed-loop version or a 30-45 page award-feel version;
- conclusion answers each task with numbers or decisions.

## Handoff Standard

Before the user tests the run in Claude Code, the repository should provide:

- updated prompt flow;
- soft-rule knowledge files;
- online-consensus template;
- team-like execution rules;
- human-feel review gate;
- no new dependency requirement for the basic workflow.
