# v1.5 Method Route Contract

Purpose: prevent v1.5 papers from looking like barely passing drafts with one shallow method per question.

This contract connects problem-family routing, method depth, evidence artifacts, and paper sections. It does not reward algorithm-name stacking. It rewards route depth that a reviewer can see.

## Required Reading Before Route Lock

Read these before fixing `runs/current/route-decision.md` and `runs/current/method-depth-plan.md`:

- `knowledge/cumcm/national-problem-family-methodology-matrix.md`
- `knowledge/algorithms/route-selection-protocol.md`
- `knowledge/algorithms/method-depth-ladder.md`
- `knowledge/cumcm/official-paper-paradigms.md`

## Minimum v1.5 Route Shape

For each major subquestion, the route must name:

```text
problem signal
-> support layer
-> main model
-> result artifact
-> validation or comparison layer
-> paper interpretation
```

Examples:

- forecast task: exploration -> baseline forecast -> improved forecast -> error check -> downstream decision;
- evaluation task: indicator system -> weighting/ranking -> sensitivity -> recommendation or planning bridge;
- optimization task: object and constraints -> feasible baseline -> improved plan -> feasibility audit -> scenario check;
- classification task: feature diagnosis -> baseline classifier -> model comparison -> threshold rule -> confusion/error analysis;
- geometry task: object figure -> coordinate frame -> derivation -> numeric result -> residual or physical-feasibility check.

## Depth Gate

Use `knowledge/algorithms/method-depth-ladder.md`.

Pass target:

- ordinary supporting subquestions: Level 3 or above;
- central decision subquestions: Level 4 when data and time permit;
- if a subquestion remains Level 2, record why a deeper route is not credible.

Fail if:

- a major subquestion is only `method -> result`;
- a final recommendation has no feasibility, sensitivity, baseline, or sanity check;
- a complex algorithm is named but its route role is unclear;
- the route cannot name the exact figure/table that will prove the result.

## Paper Evidence Gate

Before writing `paper/main.tex`, `runs/current/method-depth-plan.md` must contain a table with these columns:

| Subquestion | Family | Support layer | Main model | Result artifact | Validation artifact | Paper section | Depth |
|---|---|---|---|---|---|---|---|

Every row needs a real artifact target, such as:

- `paper/tables/q1_indicator_weights.tex`;
- `paper/figures/q2_forecast_backtest.png`;
- `paper/tables/q3_feasibility_audit.tex`;
- `paper/figures/q4_confusion_matrix.png`.

Placeholders such as `figure TBD`, `table TBD`, `some validation`, or `model result` do not pass.

Executable support:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-v1.5-pdf.ps1 -Pdf .\paper\main.pdf -Run .\runs\current -Tex .\paper\main.tex -Review .\reviews\review-current.md
```

When `-Run` is provided, the checker also inspects `method-depth-plan.md` for concrete result and validation artifacts.

## Award-Paper Feel Signals

The method route should create at least one of these visible strengths:

- baseline-to-improvement comparison;
- sensitivity or scenario analysis tied to a decision;
- feasibility audit for an executable plan;
- error analysis that changes how the conclusion is worded;
- object-first schematic before formulas;
- route bridge where one subquestion's result becomes the next subquestion's input.

Do not fake depth by adding unrelated methods. A linked two-method chain is stronger than four disconnected algorithm names.

## Abstract And Conclusion Link

The abstract may only claim results that appear in:

- the result artifact column of `method-depth-plan.md`;
- the artifact ledger;
- the body table/figure;
- the final review evidence.

If the abstract says a model is stable, accurate, optimal, feasible, or robust, the validation artifact column must identify the evidence.

## Review Gate

The final review should include:

```text
Method Route Verdict
```

with:

| Subquestion | Depth Pass/Fail | Evidence artifact | Validation artifact | Repair |
|---|---|---|---|---|

Any failed central subquestion means the paper is not ready for v1.5 handoff.

Best use:

- read after front-matter rhythm rules and before implementation;
- use together with `knowledge/quality/v1-5-hard-gates.md`;
- use when a draft looks complete but method depth feels ordinary.
