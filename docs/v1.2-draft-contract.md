# v1.2 Draft Contract

Purpose: turn the soft v1.2 goal into a concrete drafting contract that Claude Code can actually follow during generation and revision.

This note sits between the broad runbook and the detailed review rubric.

## When To Use

Read this before:

- filling `runs/current/section-budget.md`;
- filling `runs/current/writing-style-plan.md`;
- drafting `paper/sections/*.tex`;
- deciding whether a paper is still too thin for v1.2.

## Core Standard

v1.2 should look like a serious human-team draft, not a padded demo and not a method-name outline.

The contract is:

```text
stable question map
-> visible section budget
-> method chain per subquestion
-> figure/table evidence per subquestion
-> explicit validation layer
-> abstract and conclusion closure
```

If one of these links is missing, the paper is not yet at v1.2.

## Structural Targets

Use these as default targets unless the problem family genuinely requires a different split.

### Overall Length

- main body target: about 16-24 pages before appendix
- appendix target: about 3-8 pages
- full paper target: about 20-30 pages including references and appendix

The point is not page count by itself. The point is whether the structure can naturally reach this range through real content.

### Abstract

- target length: about one dense page
- expected content:
  - one opening sentence on the global task;
  - one method-result closure per major subquestion;
  - one sentence on validation, comparison, or robustness;
  - one final recommendation or pattern statement.

If the abstract stops at a short half page while the body has multiple subquestions, treat that as a likely v1.2 failure.

### Body Rhythm

Recommended body rhythm:

- problem restatement and route overview: 1-2 pages
- problem analysis: 2-4 pages
- assumptions and symbols: 1-2 pages
- data processing and exploratory evidence: 1.5-3 pages
- model establishment and solution process: 6-10 pages
- results: 2-4 pages
- validation and sensitivity: 2-4 pages
- strengths, limitations, conclusion: 1.5-3 pages

The model establishment block should usually be the longest technical center.

## Subquestion Contract

For each major subquestion, plan a full loop:

```text
analysis
-> model
-> executable artifact
-> interpretation
-> validation/comparison
-> close-out sentence
```

Minimum expectations per major subquestion:

- at least one named model or algorithm layer;
- at least one visible result artifact;
- at least one visible validation or comparison artifact when the route reasonably allows it;
- at least one table or one figure, and preferably both;
- at least one paragraph that states what conclusion the artifacts support.

If a subquestion ends at `method -> result`, it is still too shallow.

## Visual Evidence Contract

The paper should not hide all visuals in the late results section.

Default target:

- one early object-identification or route figure;
- one evidence artifact in each major subquestion block;
- one validation artifact for each major result family.

Preferred distribution:

- early figure: object, structure, route, network, geometry, or workflow;
- middle figures/tables: model inputs and main outputs;
- late figures/tables: validation, scenario comparison, sensitivity, feasibility, or ablation.

## Writing Contract

v1.2 body writing should be paragraph-driven.

Target habits:

- explain why the method fits before presenting equations;
- interpret tables and figures immediately after citation;
- use bullets mainly for assumptions, symbols, or short inventories;
- close each subquestion before moving on;
- state limitations calmly and specifically.

Risk smells:

- repeated `first/second/third` scaffolding;
- many one-sentence paragraphs in a row;
- large bullet stacks in main analytical sections;
- paragraphs that announce an artifact without interpreting it.

## Review Contract

Do not claim v1.2 readiness unless all of the following are true:

- `runs/current/method-depth-plan.md` is completed;
- `runs/current/section-budget.md` is completed;
- `runs/current/writing-style-plan.md` is completed;
- abstract closes each subquestion;
- every major subquestion has evidence plus validation/comparison coverage;
- the review does not still say the paper is thin, shallow, or machine-like;
- the machine gate passes or the open issues are explicitly justified.
