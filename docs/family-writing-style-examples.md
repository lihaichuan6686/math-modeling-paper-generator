# Family Writing Style Examples

Purpose: provide ready-to-adapt `writing-style-plan.md` examples for high-frequency CUMCM families.

Use this after route selection and before filling `runs/current/writing-style-plan.md`.

These are examples meant to make the draft sound more like a deliberate team paper and less like a scaffold.

## Example 1: Evaluation To Planning

### Style Target

- target version: `v1.2`
- route family: `evaluation to planning`
- selected style variant: `official concise closure + evidence-dense research style`
- positive sample: `2021C-comparison-C066-C085-C169-C283.md`
- contrast sample: `official-style-vs-modern-draft-risk.md`

### Core Writing Choices

- abstract density target: `one dense page with score -> plan -> audit closure`
- paragraph-driven vs bullet-driven balance: `strongly paragraph-driven in body; tables carry list-like content`
- tone target: `calm, decision-oriented, traceable`
- how results will be interpreted in prose: `every table should answer what choice is made and why`
- how limitations will be stated: `tie each limitation to missing scenario breadth or weight uncertainty`
- family-specific abstract skeleton: `evaluation route -> planning result -> feasibility or scenario check -> recommendation`
- preferred subquestion close-out pattern: `ranking meaning -> plan implication -> audit confirmation -> next-step input`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | closes score, plan, and validation | ending at ranking |
| Problem analysis | explains why ranking alone is insufficient | generic `we choose method X` |
| Model establishment | bridge from score layer to decision layer is explicit | disconnected models |
| Results | each artifact advances the executable recommendation | stacked tables without link |
| Validation | feasibility and scenario language stays visible | token robustness sentence |
| Conclusion | concise, numeric, actionable | soft qualitative wrap-up only |

### Human-Team Soft Rules

- the evaluation section should feel shorter than the planning section;
- the first strong result artifact should make the planning input concrete;
- each subquestion should end by stating what is passed to the next layer;
- sensitivity language should sound like operational caution, not textbook ceremony.

### Thinness Risks

- sections likely to become too short: `validation`
- sections likely to become too bullet-heavy: `problem analysis`
- sections likely to overuse generic prose: `strengths and limitations`
- subquestions most likely to end without a real close-out sentence: `candidate-screening subsection`

## Example 2: Forecast To Decision

### Style Target

- target version: `v1.2`
- route family: `forecast to decision`
- selected style variant: `route-heavy analytical + evidence-dense research style`
- positive sample: `e-route-comparison-2022-2023.md`
- contrast sample: `official-style-vs-modern-draft-risk.md`

### Core Writing Choices

- abstract density target: `one dense page with forecast -> decision -> uncertainty closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven with figures carrying temporal comparisons`
- tone target: `measured, uncertainty-aware, explanatory`
- how results will be interpreted in prose: `every forecast output should be tied to a downstream decision implication`
- how limitations will be stated: `name uncertainty range, data horizon, or scenario incompleteness directly`
- family-specific abstract skeleton: `trend diagnosis -> forecast -> decision -> scenario or residual validation`
- preferred subquestion close-out pattern: `forecast meaning -> decision impact -> validation caveat -> next-step transition`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | names both the prediction and what it changes | forecast-only summary |
| Problem analysis | makes uncertainty burden explicit | treating prediction as deterministic setup |
| Model establishment | forecast and decision layers are distinct but linked | one compressed model block |
| Results | forecast figure followed by decision interpretation | separate result islands |
| Validation | residual/backtest and scenario language both visible | only one fit metric |
| Conclusion | ends with strategy under uncertainty | certainty overreach |

### Human-Team Soft Rules

- trend analysis should feel purposeful, not decorative;
- the decision section should sound like a consequence of the forecast, not a separate paper;
- scenario language should be written in practical terms;
- each subquestion ending should say how uncertainty is carried forward.

### Thinness Risks

- sections likely to become too short: `decision interpretation`
- sections likely to become too bullet-heavy: `data processing`
- sections likely to overuse generic prose: `conclusion`
- subquestions most likely to end without a real close-out sentence: `forecast subsection`

## Example 3: Geometry / Spatial Design

- target version: `v1.2`

- positive sample: `spatial-measurement-comparison-B030-B086-B311.md`

### Core Writing Choices

- abstract density target: `dense page with scene -> derivation -> design -> feasibility closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven around equations, with lists moved to tables`
- tone target: `precise, scene-aware, technically calm`
- how results will be interpreted in prose: `translate formulas into visible design meaning immediately`
- how limitations will be stated: `state geometric assumptions and scene simplifications plainly`
- family-specific abstract skeleton: `scene setup -> geometric route -> computed design -> replay or feasibility`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | names the visible design result | ending with formulas only |

| Results | design/location outputs are visually concrete | one final number with no scene meaning |
| Validation | replay/overlap/residual language is visible | symbolic-only validation |
| Conclusion | returns to engineering meaning | pure mathematical summary |

### Human-Team Soft Rules

- the first figure should make the scene immediately legible;
- the symbol world should stay stable through all subquestions;
- each subquestion ending should say what physical conclusion has been obtained;
- validation should feel like a scene replay, not a checkbox.

### Thinness Risks

- sections likely to become too short: `result interpretation`
- sections likely to become too bullet-heavy: `assumptions`
- sections likely to overuse generic prose: `limitations`
- subquestions most likely to end without a real close-out sentence: `middle derivation subsections`

## Example 4: Rail / Timetable Service Planning

### Style Target

- target version: `v1.2`
- route family: `rail / timetable service planning`
- selected style variant: `route-heavy analytical + evidence-dense research style`
- positive sample: `route-example-matrix.md`
- contrast sample: `mathorcup-2023B-v1-test-review.md`

### Core Writing Choices

- abstract density target: `one dense page with flow -> service pattern -> timetable -> audit closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven with tables carrying operational details`
- tone target: `operational, quantitative, feasibility-aware`
- how results will be interpreted in prose: `each plan/timetable artifact should state service effect and feasibility meaning`
- how limitations will be stated: `state demand simplifications, recurrence limits, and scenario boundaries directly`
- family-specific abstract skeleton: `passenger-flow reading -> selected service pattern -> timetable result -> capacity/tracking/dwell audit`
- preferred subquestion close-out pattern: `service outcome -> timetable implication -> audit confirmation -> operating recommendation`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | closes service pattern and feasibility together | timetable mention without audit |
| Problem analysis | explains why the operation plan must drive the timetable | timetable-first writing |
| Model establishment | operation-pattern logic and recurrence logic are both explicit | cost function only |
| Results | service plan and timetable are both interpreted | artifact cluster at the end |
| Validation | capacity, tracking, dwell, and scenario language all appear | one summary sentence |
| Conclusion | practical service recommendation | empty policy prose |

### Human-Team Soft Rules

- passenger-flow logic should appear before recurrence equations become heavy;
- the timetable should feel like a consequence of the operation plan;
- every audit table should have one interpretive paragraph;
- subquestion endings should sound like operating decisions, not just computed outputs.

### Thinness Risks

- sections likely to become too short: `operation-to-timetable bridge`
- sections likely to become too bullet-heavy: `problem analysis`
- sections likely to overuse generic prose: `recommendation section`
- subquestions most likely to end without a real close-out sentence: `service-pattern subsection`

## Example 5: Dynamic Scheduling / Update

### Style Target

- target version: `v1.2`
- route family: `dynamic scheduling / update`
- selected style variant: `local awarded-paper dense style + evidence-dense research style`
- positive sample: `deep-reading-2021D017.md`
- contrast sample: `official-style-vs-modern-draft-risk.md`

### Core Writing Choices

- abstract density target: `one dense page with baseline -> update -> after-update effect closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven, with schedules and scenario details carried by tables and figures`
- tone target: `operational, comparative, disturbance-aware`
- how results will be interpreted in prose: `every schedule artifact should say what changed and why it matters`
- how limitations will be stated: `state disturbance scope, rolling-window assumptions, and scenario boundaries directly`
- family-specific abstract skeleton: `baseline plan -> update method -> adjusted result -> comparison or audit`
- preferred subquestion close-out pattern: `baseline/update result -> operational meaning -> audit confirmation -> next-step trigger`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | names both the baseline and the revised outcome | static-plan summary only |
| Problem analysis | separates normal and abnormal tasks clearly | mixed task narration |
| Model establishment | baseline and update logic are both explicit | update rule hidden in prose |
| Results | before/after comparison is visible and interpreted | single schedule dump |
| Validation | feasibility and stress language both appear | generic robustness sentence |
| Conclusion | states operational improvement or preservation | vague “good performance” claim |

### Human-Team Soft Rules

- the update trigger should feel concrete, not ceremonial;
- the adjusted plan should sound like a response to disturbance, not a second unrelated plan;
- each subquestion ending should say what was preserved, improved, or sacrificed;
- comparison language should be practical and numeric.

### Thinness Risks

- sections likely to become too short: `abnormal-scenario comparison`
- sections likely to become too bullet-heavy: `solution process`
- sections likely to overuse generic prose: `strengths and limitations`
- subquestions most likely to end without a real close-out sentence: `baseline schedule subsection`

## Example 6: Classification / Recognition

### Style Target

- target version: `v1.2`
- route family: `classification / recognition`
- selected style variant: `official concise closure + evidence-dense research style`
- positive sample: `deep-reading-2021E014.md`
- contrast sample: `official-style-vs-modern-draft-risk.md`

### Core Writing Choices

- abstract density target: `dense page with preprocessing -> classifier -> class-level validation closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven, with feature and metric detail carried by tables`
- tone target: `careful, diagnostic, class-aware`
- how results will be interpreted in prose: `each metric artifact should say what it means at class level`
- how limitations will be stated: `state sample limitations, class imbalance, and generalization boundaries directly`
- family-specific abstract skeleton: `preprocessing / feature route -> classifier result -> confusion/error validation`
- preferred subquestion close-out pattern: `feature/classifier result -> recognition meaning -> error check -> next-step selection`

### Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | names recognition meaning, not just metrics | accuracy-only summary |
| Problem analysis | explains why preprocessing and class-level evidence matter | generic ML wording |
| Model establishment | model comparison criteria are explicit | one-model shortcut |
| Results | confusion-level interpretation is visible | metrics dumped without prose |
| Validation | split policy, leakage, and error analysis all appear | validation by one score |
| Conclusion | closes with diagnostic implication and caveat | only repeating best accuracy |

### Human-Team Soft Rules

- preprocessing should sound like part of the model, not setup trivia;
- the chosen classifier should be justified in terms of class behavior, not only one global score;
- each subquestion ending should state what recognition conclusion is now defensible;
- limitations should sound like honest sample or feature boundaries.

### Thinness Risks

- sections likely to become too short: `error interpretation`
- sections likely to become too bullet-heavy: `data processing`
- sections likely to overuse generic prose: `conclusion`
- subquestions most likely to end without a real close-out sentence: `classifier comparison subsection`
