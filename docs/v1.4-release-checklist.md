## Static Quality Checks

- [ ] Community rules are written as reusable modeling-paper guidance, not as niche-topic notes.
- [ ] Online consensus is framed as reflection and sanity checking, not answer copying.
- [ ] Online consensus notes must include source-quality labels, route reflection, rejected/discounted signals, and remaining uncertainty.
- [ ] Online consensus notes convert useful public signals into run-file changes or explicitly reject them.
- [ ] Public community sources are used as interpretation/trap/validation signals, not as copied solutions or popularity proof.
- [ ] Public experience and writing soft rules are searched before exact problem discussion.
- [ ] Paper-feel rules emphasize abstract density, answer visibility, figure/table interpretation, and human-team rhythm.
- [ ] Team execution rules require problem profiling, route choice, code artifacts, writing rhythm, and final abstract rewrite to appear coordinated.
- [ ] Abstract/closeout rules require method-result closure, near-page-one density, and agreement across abstract, body, conclusion, and artifact ledger.
- [ ] Section-duty rules make title, abstract, keywords, restatement, analysis, assumptions, symbols, model solution, result validation, evaluation, references, and appendix each carry a visible job.
- [ ] Visual evidence-chain rules require source, role, subquestion link, caption, nearby interpretation, and body-vs-appendix boundary for important figures/tables.
- [ ] Section-rhythm rules emphasize each section's job, the one-page abstract ceiling, and evidence-driven length.
- [ ] Recent award-paper visual rhythm is used to distinguish a 20-30 page early draft from a stronger 30-45 page award-feel draft.
- [ ] Recent award-paper visual rhythm checks that pages 4-6 contain early model evidence and pages 8-12 avoid pure-prose drift when the problem supports artifacts.
- [ ] Appendix/code pages are treated as reproducibility support, not as a substitute for body formulas, result tables, figures, and interpretation.
- [ ] Section-budget rules separate effective body pages, references, appendix pages, and code appendix pages.
- [ ] References are required to support method, data, and domain claims instead of padding the bibliography.
- [ ] Literature search is framed as method/data/domain/validation support, not exact-answer hunting.
- [ ] Literature notes remain separate from online consensus notes.
- [ ] Source-quality rules reject paid leaks, private materials, and answer-only dumps.
- [ ] v1.4 notes preserve the research-only and responsible-use boundary.
- [ ] Remaining local `.doc` and `.ppt` absorption has a priority queue, safe extraction protocol, and completion markers.
- [ ] Old-format extraction attempts can be marked partial or blocked instead of being counted as full learning.
- [ ] Old binary Office probes use a repeatable script instead of one-off manual scanning when possible.

## Ready For User Test When

The repository is ready for a v1.4 user-side Claude Code test when:

1. all required files exist;
2. all entry points mention the v1.4 workflow;
3. a new run scaffold includes `online-consensus-notes.md`;
4. the quality prompt checks v1.4 human-feel and online-consensus behavior;
5. no single rare topic is treated as a core route family.

## Not Required For v1.4 Static Release

- proving that one generated paper wins;
- running a full end-to-end paper production test locally;
- optimizing one historical problem;
- adding heavy dependencies.
