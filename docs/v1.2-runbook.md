# v1.2 Runbook

Purpose: give Claude Code a direct path to a stronger paper that reads like a serious human team draft rather than a thin demo.

This is the upgrade path after v1.0 proved the loop.

## v1.2 Philosophy

v1.2 is not "slightly bigger v1.0".

Its job is to repair the main weak points of v1.0:

- paper length that is too short or uneven;
- prose that is structurally complete but too thin;
- methods that stop at a shallow single-model answer;
- section and bullet structure that feels machine-like instead of team-written;
- abstract, analysis, results, and conclusion that do not yet carry enough contest-grade density.

The v1.2 target is:

```text
human-team paper feel
-> stronger section density
-> deeper method chain
-> more natural paragraph flow
-> visible validation and comparison
-> 20-30 page shape without filler
```

## Human-Team Paper Rule

The reader should feel that a strong team worked through the problem in layers:

```text
what the task asks
-> why this route fits
-> how the model is built
-> what the computed answer is
-> why the answer is credible
-> where the method still has limits
```

If the paper jumps too quickly from method name to final table, it is still below v1.2.

## Method Depth Rule

For each major subquestion, avoid a shallow one-step chain.

Prefer this structure:

```text
problem signal
-> primary model
-> supporting method or preprocessing layer
-> result artifact
-> validation / sensitivity / baseline comparison
```

Examples:

- ranking only -> too shallow
- ranking -> executable plan -> feasibility audit -> sensitivity table -> v1.2-ready

- forecast only -> too shallow
- forecast -> decision model -> scenario comparison -> service or policy consequence -> v1.2-ready

## Section Density Rule

The paper should grow through section roles, not through repeated filler.

Expected density:

- abstract: one dense page, with method and result closure for each subquestion;
- problem analysis: not less than the route needs, usually 2-4 pages with dependency or route artifacts;
- model establishment: the largest section, because this is where mathematical substance lives;
- results and validation: must both be visible and cannot be merged into one thin paragraph;
- conclusion: compact, but still answers each subquestion and states limits.

Use `runs/current/section-budget.md` to control this before drafting.
Use `docs/v1.2-draft-contract.md` when section targets still feel too vague.

## Writing Style Rule

v1.2 should sound like a team paper, not like generated notes.

Required writing habits:

- use compact analytical paragraphs instead of repeated bullet lists;
- explain why a method is used before showing equations;
- interpret each table and figure immediately after it appears;
- let each subquestion close a full loop before moving on;
- keep abstract, section titles, tables, and conclusion on one stable question map.

Use `runs/current/writing-style-plan.md` to lock the intended voice and density before section drafting.
Use `docs/v1.2-draft-contract.md` when the draft still needs concrete per-subquestion closure and evidence targets.

## v1.2 Acceptance Gate

A run is not v1.2-ready unless:

- the section budget exists and is followed;
- the abstract is intentionally written to near one dense page;
- every subquestion has a method chain, visible result, and visible validation or comparison;
- the main method is not standing alone without a support or check layer;
- the prose no longer reads like stacked bullet fragments;
- the paper shape can plausibly reach contest-grade length from real content;
- the review explicitly checks for machine-like thinness, unsupported method names, and awkward section proportions.

If those checks still fail after the first draft and review, do not accept a cosmetic pass. Run `prompts/09_revision_v1_2.md` and repair the flagged weak points in the paper sections and run artifacts before the next review.

Before treating a v1.2 draft as handoff-ready, also run the shorter pass in `docs/v1.2-final-pass-guide.md` so question-map stability, evidence distribution, appendix boundary, and PDF finish are checked together.
