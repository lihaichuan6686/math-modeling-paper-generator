# Problem Profile

Run: example-geometry-spatial

## Problem Image

This is a scene-driven design problem. The paper must first make the geometric object legible, then derive the model, and finally close on a feasible design or positioning result with visible physical meaning.

## Task Family And Route Signals

- primary family: geometry / spatial design
- support family: feasibility comparison
- route signals: coordinates, distances, angles, coverage or location constraints, final configuration required

## Modeled Object

The modeled object is a geometric or physical scene with explicit coordinates, structural relations, and feasibility constraints.

## Decision Object

The final decision object is a feasible design, location, parameter set, or coverage configuration, not only an equation result.

## Question Map

- Q1: define the scene and derive the coordinate or geometric relations
- Q2: compute the target design or positioning result
- Q3: compare or validate the configuration under the stated constraints

## Evidence Burden

- a symbolic derivation alone is insufficient
- the paper needs a readable scene artifact
- the final design must be shown numerically or visually
- feasibility or replay evidence is required

## Visual Burden

- one early scene schematic
- one equation or parameter summary artifact
- one design/result artifact
- one replay, coverage, overlap, or feasibility artifact

## Thinness Risks

- Q1 may be too short if the scene is not explained before formulas start
- Q2 may become equation-heavy without physical interpretation
- Q3 may shrink into one residual number without design meaning

## Fake-Completion Risks

- optimizing before the object is clearly defined
- ending with formulas instead of a visible configuration
- claiming engineering readiness from contest-level assumptions only
