# Writing Style Plan

## Style Target

- target version: `v1.2`
- route family: `rail / timetable service planning`
- selected style variant: `route-heavy analytical + evidence-dense research style`
- positive sample: `route-example-matrix.md`
- contrast sample: `mathorcup-2023B-v1-test-review.md`

## Core Writing Choices

- abstract density target: `one dense page with flow -> service pattern -> timetable -> audit closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven, with operational details moved into tables`
- tone target: `operational, quantitative, feasibility-aware`
- how results will be interpreted in prose: `each plan/timetable artifact should state service effect and feasibility meaning`
- how limitations will be stated: `state demand aggregation limits, recurrence limits, and scenario boundaries directly`
- family-specific abstract skeleton: `flow reading -> selected service pattern -> timetable result -> capacity/tracking/dwell audit`
- preferred subquestion close-out pattern: `service outcome -> timetable implication -> audit confirmation -> operating recommendation`

## Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | closes service pattern and feasibility together | timetable mention without audit |
| Problem analysis | explains why the operation plan must drive the timetable | timetable-first writing |
| Model establishment | operation-pattern and recurrence logic are both explicit | cost function only |
| Results | service plan and timetable are both interpreted | artifact cluster at the end |
| Validation | capacity, tracking, dwell, and scenario language all appear | one summary sentence |
| Conclusion | practical service recommendation | empty policy prose |

## Human-Team Soft Rules

- passenger-flow logic should appear before recurrence equations become heavy;
- the timetable should feel like a consequence of the selected operation plan;
- every audit table should have one interpretive paragraph;
- subquestion endings should sound like operating decisions, not just computed outputs.

## Thinness Risks

- sections likely to become too short: `operation-to-timetable bridge`
- sections likely to become too bullet-heavy: `problem analysis`
- sections likely to overuse generic prose: `recommendation section`
- subquestions most likely to end without a real close-out sentence: `service-pattern subsection`
