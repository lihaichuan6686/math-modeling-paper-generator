# Writing Style Plan

## Style Target

- target version: `v1.2`
- route family: `forecast to decision`
- selected style variant: `route-heavy analytical + evidence-dense research style`
- positive sample: `e-route-comparison-2022-2023.md`
- contrast sample: `official-style-vs-modern-draft-risk.md`

## Core Writing Choices

- abstract density target: `one dense page with forecast -> decision -> uncertainty closure`
- paragraph-driven vs bullet-driven balance: `paragraph-driven in body; temporal detail moved into tables and figures`
- tone target: `measured, explanatory, uncertainty-aware`
- how results will be interpreted in prose: `every forecast artifact should say what decision it changes`
- how limitations will be stated: `name data horizon, scenario range, and uncertainty scope directly`
- family-specific abstract skeleton: `trend diagnosis -> forecast -> decision -> scenario or residual validation`
- preferred subquestion close-out pattern: `forecast meaning -> decision impact -> validation caveat -> next-step transition`
- family-specific density cue: `the forecast-to-decision bridge and uncertainty explanation must be heavier than background trend narration`

## Section Notes

| Section | Intended feel | What must not happen |
| --- | --- | --- |
| Abstract | names both prediction and what it changes | forecast-only summary |
| Problem analysis | makes uncertainty burden explicit | deterministic setup prose |
| Model establishment | forecast and decision layers are distinct but linked | one compressed model block |
| Results | forecast artifact followed by decision interpretation | separate result islands |
| Validation | residual/backtest and scenario language both visible | only one fit metric |
| Conclusion | ends with strategy under uncertainty | certainty overreach |

## Paragraph Rhythm

- preferred rhythm: `trend finding -> forecast implication -> decision consequence -> transition`
- maximum tolerance for stacked micro-paragraphs: `avoid long runs of one-sentence trend comments`
- list-risk zones: `data processing and conclusion`
- required interpretation zones: `after forecast artifact`, `after decision artifact`, `after scenario comparison`
- family-specific bridge paragraphs that must exist: `forecast output to decision parameter paragraph`, `uncertainty propagation paragraph`
- section most likely to become artifact-stacked: `results`
- section that should stay intentionally compact: `problem restatement`

## Human-Team Soft Rules

- the decision section should read like a consequence of the forecast, not a separate paper;
- uncertainty language should sound operational rather than ceremonial;
- each subquestion should end by stating what information is carried forward;
- recommendation language should stay scoped to the tested horizon and scenarios.

## Thinness Risks

- sections likely to become too short: `decision interpretation`
- sections likely to become too bullet-heavy: `data processing`
- sections likely to overuse generic prose: `conclusion`
- subquestions most likely to end without a real close-out sentence: `forecast subsection`
