# v1.4 User Test Protocol

Purpose: give the user a consistent way to test a v1.4 Claude Code run and report whether the paper feels closer to a serious CUMCM team paper.

This protocol is for user-side practice tests. It does not prove award quality.

## Test Setup

Use a fresh problem and the direct v1.4 launch prompt:

```text
prompts/12_launch_v1_4.md
```

When public-web access is available, the run should use `prompts/13_platform_research.md` for Playwright MCP public research before `prompts/11_online_consensus_check.md`.

Before starting, prepare:

- problem statement;
- attachments or data files;
- any official template requirement;
- optional excellent-paper comparison sample if available.

## Required Run Artifacts

After the run, check that these exist:

- `runs/current/problem-profile.md`
- `runs/current/route-decision.md`
- `runs/current/model-plan.md`
- `runs/current/online-consensus-notes.md`
- `runs/current/figure-plan.md`
- `runs/current/section-budget.md`
- `runs/current/writing-style-plan.md`
- `runs/current/artifact-ledger.md`
- `paper/main.tex`
- `paper/main.pdf` if compilation is available
- `reviews/review-current.md`

## Test Questions

### 1. First-Look Paper Feel

Check the PDF before reading details:

- Does page 1 look like title plus dense abstract and keywords?
- Does page 2 enter problem restatement or analysis quickly?
- Do pages 4-6 show model evidence such as symbols, equations, model diagram, data table, or parameter table?
- Do pages 8-12 already alternate artifacts and interpretation?
- Does the middle body avoid long pure-prose stretches?
- Does appendix code come after the body has already explained the model and results?

### 2. Content Strength

Check:

- Are all subquestions answered?
- Does each major subquestion have model, result, and validation?
- Are final answer tables visible in the body?
- Are figures/tables interpreted immediately?
- Are references real and tied to method/data/domain claims?
- Does `literature-notes.md` separate literature support from online consensus?
- Did literature search change run files or citation duties instead of hunting answers?
- Does the paper explain whether it is a 20-30 page early draft or a 30-45 page award-feel draft?

### 3. Online Consensus Behavior

Check `online-consensus-notes.md`:

- Were Playwright MCP public browsing attempts, searched queries, or platform attempts recorded?
- Were inaccessible login/captcha/paywalled/private/app-only pages recorded and skipped rather than bypassed?
- Are source-quality labels present?
- Is there route reflection?
- Are weak/unsafe signals rejected or discounted?
- Is remaining uncertainty recorded?
- Did the paper avoid copying online answers?

### 4. Human-Team Feel

Check whether the paper feels like a team worked through the problem:

- route choice is explained before method names;
- simple baseline or comparison appears when useful;
- limitations are concrete;
- paragraphs do not read like a template;
- the conclusion repeats actual answers rather than praising the model.

## Scoring

Use this quick score:

| Area | Score 0-2 | Meaning |
|---|---:|---|
| First-look award feel | | 0 weak, 1 usable, 2 strong |
| Method depth | | 0 shallow, 1 acceptable, 2 strong |
| Figures/tables | | 0 decorative/missing, 1 usable, 2 evidence-bearing |
| Online consensus | | 0 missing/unsafe, 1 basic, 2 reflective |
| Section proportion | | 0 padded/unbalanced, 1 acceptable, 2 deliberate |
| Human-team prose | | 0 machine-like, 1 mixed, 2 credible |

Suggested interpretation:

- 0-5: not v1.4-ready;
- 6-8: usable but needs targeted repair;
- 9-12: good v1.4 test result.

## What To Report Back

Use `docs/v1.4-user-test-feedback-template.md`.

The most useful feedback is not "good/bad"; it is:

- which v1.4 rule failed;
- where in the paper it failed;
- what the generated run files claimed;
- whether the review caught the issue.

## Not Required

For v1.4 testing, do not require:

- proving the paper would win;
- comparing against every available excellent paper;
- full manual rerun of every code file;
- perfect OCR/reference statistics.

The point is to test whether v1.4 rules visibly improve the generated paper.
