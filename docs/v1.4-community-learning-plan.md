# v1.4 Community Learning Plan

Purpose: turn contest-circle experience, public discussion, and local award-paper observations into reusable behavior for the paper generator.

This plan follows the v1.4 direction: improve the whole method, not one narrow problem type.

## Goal

Make the generator behave more like a trained CUMCM team:

- it knows the soft rules of the Chinese mathematical modeling circle;
- it uses public discussion to avoid blind spots without copying answers;
- it writes with award-paper section rhythm, not template padding;
- it treats figures, tables, validation, and abstract density as visible judging surfaces.

## Track 1: Local Experience Absorption

Primary folder:

```text
D:\2-各比赛Word模板及LaTeX模板\9.数学建模经验分享（36篇）
```

Read priority:

1. paper-writing experience notes;
2. contest attention and writing training;
3. personal modeling reflections;
4. newcomer tutorials and team-work notes;
5. PPT/DOC training files that describe time allocation, section structure, and common mistakes.

Extract only reusable rules:

- section duty: what each paper section must visibly accomplish;
- abstract duty: what method/result information must appear on page one;
- route duty: how to avoid forcing the task into a familiar method;
- evidence duty: how figures, tables, and validation support claims;
- review duty: what makes a paper look thin, copied, or machine-like;
- collaboration duty: how a team would split modeling, coding, and writing.

Convert signals through `docs/community-signal-to-rule-pipeline.md`.
Do not mark a source as absorbed until its useful signals have been converted into a run-file rule, prompt requirement, review gate, or stable knowledge note.

Use `docs/local-experience-extraction-queue.md` to choose the next `.doc`/`.ppt` batch and `docs/local-experience-extraction-status.md` to record whether each source is readable, partially readable, absorbed, or blocked.

Do not extract:

- long quoted passages;
- full solution text;
- old examples that only apply to one historical problem;
- method lists without paper-writing or route-selection consequences.

## Track 2: Public Contest-Circle Soft Rules

Target public sources:

- Bilibili public videos and public comments when accessible;
- Xiaohongshu public posts when accessible through public pages;
- Zhihu answers;
- CSDN/blog posts;
- university training notes;
- official contest pages;
- public GitHub/Gitee repos only for structure and reproducibility habits.

Search first for general experience:

```text
数学建模 国赛 经验
数学建模 论文写作 摘要
数学建模 国赛 常见错误
数学建模 获奖论文 结构
数学建模 小红书 经验分享
数学建模 B站 论文写作
```

Then, during a concrete run, search the specific problem only after the initial route exists.

Reusable soft-rule categories:

- taboo phrases and overclaims;
- abstract length and density;
- section proportion;
- common reviewer irritants;
- "looks like a strong team" signals;
- graph/table density;
- when online result comparison helps and when it creates panic.

## Track 3: Pre-Writing Online Consensus Check

Insert this step after initial model planning and before final paper writing:

```text
problem-profile
-> route-decision
-> model-plan
-> public online-consensus check
-> route reflection
-> code, figures, tables, paper
```

The check should produce:

```text
runs/current/online-consensus-notes.md
```

Required contents:

- searched queries or platform attempts;
- source quality labels;
- common interpretations;
- common route families;
- rough result ranges if repeatedly and credibly mentioned;
- traps and wording warnings;
- rejected or discounted signals;
- route changes or reasons for no change;
- remaining uncertainty.

The note must not contain copied final answers or private material.

## Track 4: Award-Paper Feel Calibration

Continue sampling local excellent papers and record:

- page count and effective body length;
- abstract density and whether it nearly fills page one;
- when problem restatement starts and ends;
- when the first substantial model artifact appears;
- figure/table frequency in the middle body;
- number and type of references;
- appendix boundary and code-listing style.

Convert observations into:

- `section-budget.md` rules;
- `figure-plan.md` expectations;
- `writing-style-plan.md` expectations;
- `v1.4-human-feel-review-gate.md` checks.

Promote observations by maturity:

- observed: keep in an absorption log;
- repeated: use as run guidance;
- stable: add to launch prompt or review gate;
- tested: keep for v1.x repair or promote to v2.0 planning.

The user-facing effect should be simple:

```text
Claude Code clones the repo
-> reads START_HERE

```

## v1.4 Acceptance Standard

The version is ready for user testing when:

- the v1.4 launch prompt explicitly requires public consensus reflection;
- the online-consensus note has quality labels and rejection logic;
- section/page rhythm rules are loaded by the launch prompt;
- the human-feel review gate can fail a paper for weak abstract, thin body, decorative figures, or appendix-inflated length;
- user test protocol exists to score the generated paper after Claude Code runs it.

v1.4 is not expected to be commercial grade.
It should be strong enough for repeated user-side tests and targeted repair.
