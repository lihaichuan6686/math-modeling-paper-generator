# v1.5 Readiness Report

Status: static test candidate.

Purpose: summarize what v1.5 is ready to test, what it does not prove, and how user-side Claude Code results should be judged.

## What v1.5 Adds

v1.5 is an award-paper-feel hard-gate layer on top of the earlier v1.0-v1.4 workflow.

It adds:

- Level 0 reading discipline so Claude Code does not burn the run reading the whole repository first;
- neutral LaTeX skeleton at `paper/templates/cumcm_v15_main.tex`;
- front-matter rhythm rules for title, abstract, page-one density, and early concept figure;
- title-candidate workflow through `runs/current/title-candidates.md`;
- method-depth route contract through `docs/v1.5-method-route-contract.md`;
- v1.5 hard gates in `knowledge/quality/v1-5-hard-gates.md`;
- executable setup through `scripts/setup.ps1`;
- executable static check through `scripts/check-v1.5-static.ps1`;
- executable PDF/source/run/review check through `scripts/check-v1.5-pdf.ps1`;
- review scaffold with `v1.5 Hard Gate Verdict` and `Method Route Verdict`.

## Failure Classes v1.5 Is Designed To Block

v1.5 is specifically designed to block these observed user-test failures:

- task-label or AI-like title;
- one-block abstract with too much blank page-one space;
- abstract without bold key results;
- missing early concept/mechanism/model-flow figure;
- prompt-planning language leaking into paper text;
- default or unreadable figures;
- shallow `method -> result` routes;
- recommendation built on obvious sample-size or sanity problems;
- missing appendix code or script index;
- unreplaced LaTeX skeleton placeholders;
- final review that contains a table but no real Pass/Fail verdict.

## What The Static Checks Prove

`scripts/check-v1.5-static.ps1` proves:

- required v1.5 files exist;
- Level 0 navigation mentions the v1.5 contracts;
- setup, template, check, and prompt entry points are wired together;
- the new run scaffold includes title, method-depth, and review gates;
- v1.4 static compatibility is not intentionally removed.

The static check does not prove that a generated paper is good.

## What The PDF/Source/Run/Review Check Proves

`scripts/check-v1.5-pdf.ps1` can inspect:

- PDF page count and page-one density;
- abstract markers in extracted PDF text;
- prompt-language leakage in PDF and LaTeX source;
- early concept figure markers;
- appendix/reference markers;
- title-candidate file completion;
- method-depth artifact and validation completion;
- LaTeX abstract bold-result count;
- unreplaced LaTeX skeleton placeholders;
- final review verdict completion.

This check is a hard screen, not a judge. It catches obvious failures; it cannot guarantee award quality.

## What v1.5 Still Does Not Prove

v1.5 does not yet prove:

- true award competitiveness;
- correctness of a model for a new problem;
- originality or cleverness of the final route;
- visual quality when figures are image-only and not text-extractable;
- exact page layout quality without human PDF inspection;
- whether public online consensus was useful or safely interpreted;
- whether the generated paper would pass any real contest policy.

## User-Side Test Standard

A v1.5 user-side test should be considered useful when it includes:

- one fresh problem statement and attachments;
- generated `paper/main.pdf`;
- generated `paper/main.tex`;
- `runs/current/title-candidates.md`;
- `runs/current/method-depth-plan.md`;
- `runs/current/artifact-ledger.md`;
- `reviews/pdf-v15-check.md`;
- `reviews/review-current.md`;
- the user's visible-paper feedback, preferably recorded in `docs/v1.5-user-test-feedback-template.md`.

## Pass Bar For A v1.5 Test Candidate

A v1.5 run is only a test candidate when:

- `scripts/check-v1.5-static.ps1` passes before handoff;
- `scripts/check-v1.5-pdf.ps1 -Pdf .\paper\main.pdf -Run .\runs\current -Tex .\paper\main.tex -Review .\reviews\review-current.md` is run after generation;
- `reviews/pdf-v15-check.md` has no FAIL item;
- `reviews/review-current.md` has no Unknown verdict rows;
- the user visually agrees that the paper no longer has the obvious v1.4/v1.5 AI-report failures.

## Next Repair Loop

After each user-side run:

1. record failures in `docs/v1.5-user-test-feedback-template.md`;
2. classify each failure with `docs/v1.5-feedback-triage-matrix.md`;
3. repair the smallest stable contract, template, prompt, checker, or knowledge note that would prevent that failure next time;
4. rerun `scripts/check-v1.5-static.ps1`;
5. do not expand v1.5 with broad new soft rules unless the failure requires it.

## Current Recommendation

Use v1.5 for the next user-side Claude Code test before starting a v1.6 or v2.0 branch.

The next improvement should come from an actual generated paper, not from more speculative polish.
