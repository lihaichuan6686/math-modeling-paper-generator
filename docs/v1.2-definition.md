# v1.2 Definition

Purpose: define what `v1.2` means for this repository, what it must achieve before user testing, and what work is explicitly deferred to later versions.

This file exists to stop the project from expanding forever without a stable test point.

## v1.2 Positioning

`v1.2` is the first version that should feel worth testing as a serious Claude Code paper workflow.

It is **not** the final commercial-grade system.

Its role is:

```text
usable direct-start workflow
-> stronger human-team paper feel
-> clearer evidence discipline
-> sharper self-review
```

## What v1.2 Must Do

Before it is declared ready for user testing, `v1.2` must satisfy all three goals below.

### Goal A: Claude Can Start Directly

Claude should be able to clone the repository, read the documented entry points, and find the strongest route-specific references without guesswork.

Required support:

- clear repo entry points;
- route-aware sample run packages;
- route bundles, claim maps, caption maps, and review cases;
- stable run scaffold and machine gate.

### Goal B: Claude Can Produce A Clearly Better Paper Than v1.0

The output should no longer look like a small demo with thin prose.

Required signs:

- abstract aims at one dense page when the task is not tiny;
- each major subquestion closes a full loop;
- figures and tables are interpreted rather than dumped;
- results and validation are both visible;
- the paper reads more like a student team draft than a generated outline.

### Goal C: Claude Can Self-Review In A Useful Way

The system should not only generate. It should also detect why a weak draft is still weak.

Required signs:

- review output distinguishes critical findings, important findings, and warnings;
- abstract/conclusion claims are checked against artifacts;
- figure/table captions and nearby prose are checked for genericness;
- route-specific failure patterns are recognized and named.

## What v1.2 Does Not Need To Do

The following are deliberately out of scope for `v1.2`:

- guaranteed award-level competitiveness;
- full coverage of every contest family;
- real-production robustness;
- full automation of all attachment parsing;
- complete absorption of all local source materials;
- polished commercial review UX;
- final daily-sync and remote-operations reliability.

Those belong to later versions.

## v1.2 Required Artifacts

The repository should already provide:

- `docs/v1.2-runbook.md`
- `docs/v1.2-draft-contract.md`
- `docs/v1.2-final-pass-guide.md`
- `docs/sample-run-packages/README.md`
- route-specific sample run packages
- route-specific demo bundles
- route-specific claim maps
- route-specific caption maps
- route-specific review case examples

For a real active run, `v1.2` also expects:

- `runs/current/problem-analysis.md`
- `runs/current/model-plan.md`
- `runs/current/method-depth-plan.md`
- `runs/current/verification-plan.md`
- `runs/current/section-budget.md`
- `runs/current/writing-style-plan.md`
- `runs/current/artifact-ledger.md`
- paper sections, figures, tables, and review output.

## Test-Ready Standard

`v1.2` is ready for the user to test when all of the following are true:

1. the start path is clear enough that Claude can find the nearest route package quickly;
2. at least one route family has a near-real sample run package with explicit manifest and artifact story;
3. the repository contains route-aware section, assembly, claim, caption, and review examples;
4. the machine gate catches placeholders and missing planning files;
5. the system can plausibly generate a full draft and also plausibly say `Needs revision` for the right reasons.

## Stop Rule

Do not delay `v1.2` forever by chasing every polish item.

Once the test-ready standard above is met, the correct action is:

```text
hand it to the user for testing
-> collect failures
-> promote the right repairs into v1.2.x or v2.0
```

## Next Version Boundary

After user testing of `v1.2`, the likely next track is:

- `v1.2.x`: bug fixes, tighter examples, better route coverage, improved real-run clarity;
- `v2.0`: stronger modeling competitiveness, more realistic data pipelines, broader route support, and more durable review automation.
