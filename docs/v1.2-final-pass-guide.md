# v1.2 Final Pass Guide

Purpose: give Claude Code one short, actionable final-pass sequence before a v1.2 draft is treated as a serious research-ready paper candidate.

This is a handoff guide, not a full review rubric.

## When To Use

Use this guide after:

- the main body is drafted;
- figures and tables exist;
- `paper/main.tex` is assembled;
- at least one review pass has happened or is about to happen.

Use it before claiming the run is ready.

## Final Pass Sequence

Run this sequence in order:

```text
question map
-> section density
-> figure/table interpretation
-> validation strength
-> appendix boundary
-> PDF finish
-> review rerun
```

## 1. Question Map Check

Confirm:

- abstract, body headings, result tables, and conclusion use the same subquestion map;
- abstract and conclusion claims can be traced back to artifact-ledger entries or equivalent generated outputs;
- no subquestion disappears in the middle of the paper;
- each major subquestion has analysis, model, result, and validation presence.

If not, repair structure before polishing language.

## 2. Section Density Check

Confirm:

- abstract is close to one dense page when the task is not tiny;
- section lengths fit the active family rather than a generic flat split;
- problem analysis explains route choice, not just task restatement;
- model establishment is the real technical center of the paper;
- results and validation are both visible;
- conclusion is short but still closes every subquestion.

If a section feels thin, add missing loop content rather than filler prose.

## 3. Figure and Table Check

Confirm:

- the first strong figure identifies the modeled object, geometry, route, or data structure;
- every important subquestion owns at least one meaningful figure or table artifact;
- captions say what is shown, why it matters, and what the reader should notice;
- every major figure or table is interpreted in nearby prose;
- filenames in the generator output are stable and traceable.

If visuals cluster only at the end, redistribute evidence into earlier subquestion sections.

## 4. Validation Check

Confirm:

- each major subquestion has a visible check layer;
- sensitivity, baseline comparison, residual, feasibility, or scenario evidence is not token-only;
- the paper does not stop at `method -> result`;
- confidence language matches the real evidence strength.

If validation is thin, repair that before expanding appendix.

## 5. Appendix Boundary Check

Confirm:

- body closure is complete before appendix code begins;
- references appear before code appendix;
- appendix note explains whether only core excerpts are shown;
- support-file inventory appears before long code excerpts;
- appendix code is grouped by subquestion and role;
- the body would still feel complete if appendix code pages were removed.

If the body depends on appendix screenshots to feel complete, move the missing logic back into the body.

## 6. PDF Finish Check

Confirm:

- no missing figures or broken references;
- no clipped tables or unreadable labels;
- no placeholder title, `xx`, `xxxx`, or stale identity metadata;
- no watermark, promotional residue, or empty tail pages in the research copy;
- appendix transition is visually clear;
- references are rendered and readable.

## 7. Review Rerun

Before final handoff:

1. rerun `prompts/06_quality_review.md`;
2. if the draft still reads thin or scaffold-like, run `prompts/09_revision_v1_2.md`;
3. rerun the machine gate and keep the result in the review.

## Stop Rule

Do not stop at:

- compile success alone;
- page count alone;
- appendix expansion alone;
- cleaner wording without structural repair.

Stop only when:

- the body closes all subquestions;
- the figures and tables feel like evidence rather than decoration;
- the validation layer is visible;
- the ending feels like a completed paper rather than an interrupted draft.
