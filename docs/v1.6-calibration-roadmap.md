# v1.6 Calibration Roadmap

Purpose: define the remaining calibration work before v1.6 is treated as a release candidate. The goal is to turn v1.6 static rules into behavior that survives several problem families.

## Version Theme

v1.6 calibration is a release-candidate hardening phase:

```text
v1.6 rules -> fresh problem runs -> failure buckets -> route/section/visual/checker repairs
```

Do not make v1.6 calibration a broad feature release unless test evidence demands it.

## Primary Goals

| Goal | Why it matters | Deliverable |
|---|---|---|
| Multi-family calibration | a rule that works on one paper may fail on another family | three filled calibration logs and one summary |
| Same-problem comparison expansion | improves award-paper feel without overfitting a rare topic | at least one new comparison note |
| Visual-generation helper decision | concept figures are central but still under-tooled | `knowledge/visuals/v1-6-visual-generation-decision.md` and `docs/concept-figure-helper.md`, then external workflow only if calibration demands it |
| Checker false-negative repair | layout and source gates should catch repeated visible defects | checker repair log and updated gate |
| Knowledge coverage tracking | prevents unbounded learning with no product effect | updated coverage audit |
| Reference rhythm calibration | references affect award-paper credibility and claim support | `knowledge/latex/v1-6-reference-and-citation-rhythm.md` and calibration feedback |
| Excellent-paper reader lens | generated sections still can look ceremonial even when page count and layout pass | `knowledge/community/v1-6-excellent-paper-reader-lens.md` and user-side first-look feedback |
| Award-paper quantity calibration | page count and quantity feel should distinguish excellent papers from solution notes and template demos | `knowledge/latex/v1-6-award-paper-quantity-calibration.md` and calibration feedback |

## Calibration Batch

Pick three families from different reasoning modes:

1. evaluation-to-planning or supplier/material planning;
2. forecast-to-decision or monitoring/policy;
3. geometry/network/engineering or classification/recognition.

For each family, produce:

- `problem-profile.md`
- `route-decision.md`
- `method-depth-plan.md`
- `section-rhythm-budget.md`
- `figure-style-spec.md`
- `artifact-ledger.md`
- `reviews/final-v16-check.md`
- a short calibration note explaining what broke.

## Repair Priority

Use this order:

1. fix failures that make papers obviously non-human at first glance;
2. fix failures that hide or weaken final answers;
3. fix failures that make methods look shallow or unsupported;
4. fix layout defects that survived v1.6 static checks;
5. add new knowledge only when it changes a route, section, visual, or gate.

## Stop Rule

Do not start v2.0 until:

- at least three v1.6 generated papers have been reviewed;
- repeated failures are classified in the feedback triage matrix;
- filled logs based on `docs/v1.6-calibration-log-template.md` exist for the reviewed papers;
- v1.6 calibration repairs have at least one passing static check;
- the user confirms the generated paper feels materially closer to a human team draft.
