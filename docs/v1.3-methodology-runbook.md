# v1.3 Methodology Runbook

Purpose: upgrade the repository from a problem-by-problem paper loop into a reusable methodology engine that behaves more like a strong human modeling team.

v1.3 does not mean "make one tested paper prettier".

Its job is to improve the general method:

```text
problem reading
-> problem profiling
-> route selection
-> model-chain planning
-> figure/table planning
-> paper composition planning
-> self-review gate
-> revision
```

This layer sits above v1.2.
v1.2 is the stronger draft loop.
v1.3 is the methodology layer that tells the loop how to think before it writes.

## What v1.3 Adds

Compared with the v1.2 loop, v1.3 adds five required controls:

1. a problem-profile step before route selection;
2. an explicit route-decision record, not just a chosen candidate;
3. a methodology checklist that ties route, evidence, visuals, and writing together;
4. a human-team paper-composition plan, not only section budgets;
5. a stronger self-review gate aimed at fake completion, weak closure, and machine-like structure.

## Required v1.3 Run Outputs

The normal v1.2 outputs still apply.
In addition, a v1.3 run must also maintain:

- `runs/current/problem-profile.md`
- `runs/current/route-decision.md`
- `runs/current/methodology-checklist.md`

These files are not optional notes. They are the bridge between the knowledge base and the paper draft.

## v1.3 Problem-Profile Rule

Before choosing models, describe the problem in these terms:

```text
task family
modeled object
decision object
evidence burden
visual burden
human-review risk
```

Minimum questions:

- What is the core object being modeled?
- What decision, judgment, or recommendation must the paper finally produce?
- What kind of evidence will convince a reviewer for this task family?
- Which sections are likely to become thin if not planned early?
- Which hidden traps may cause the paper to look complete while still missing the real task?

Use `knowledge/cumcm/problem-understanding-framework.md`.

## v1.3 Route-Decision Rule

Do not stop at "Candidate B is selected."

Record:

```text
selected route family
why it fits the question map
why the rejected routes were worse
what evidence chain this route requires
what visuals this route requires
what fake-completion risks this route creates
```

Use `knowledge/algorithms/route-selection-protocol.md`.

## v1.3 Methodology Checklist Rule

Before implementation and drafting, record:

- stable question map;
- object-first figure;
- primary model chain by subquestion;
- validation and comparison burden by subquestion;
- expected decision closure by subquestion;
- sections most at risk of sounding machine-written;
- sections most at risk of becoming too thin;
- minimum required evidence for abstract and conclusion.

This checklist is the shortest proof that the run is being guided by a reusable method rather than one-off improvisation.

## v1.3 Paper-Composition Rule

Paper quality is not only a section-length problem.

The draft should behave like a team paper:

```text
identify the object
-> explain the route
-> define the model
-> show the computed artifact
-> interpret it
-> validate it
-> close the subquestion
```

Use `knowledge/latex/human-team-paper-composition.md` to control:

- paragraph rhythm;
- placement of early figures;
- section close-out style;
- when to use comparison tables;
- when to split a large section into multiple subquestion loops.

## v1.3 Review Gate

The run cannot be considered methodology-ready unless the review checks:

- whether the paper answered the real decision task and not only an intermediate calculation;
- whether the route was chosen for fit rather than prestige;
- whether each major subquestion closes a complete loop;
- whether the object is visible before late-stage results;
- whether visuals and tables are doing argumentative work;
- whether the abstract and conclusion overclaim relative to the ledger;
- whether the structure still reads like a generated scaffold.

Use `knowledge/quality/v1-3-self-review-gate.md` together with the normal v1.2 review files.

## Best Use

Use this runbook when:

- the user wants a stronger general workflow rather than a single patched paper;
- Claude keeps solving tasks but not building a stable reusable method;
- drafts are structurally complete but still feel machine-made;
- route selection is drifting or overfitting to one demo problem.
